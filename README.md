# КастрюляМаркет — Контрольная работа №3

Краткое описание того, что делает каждая практика и как она реализована в проекте.

---

## ПР13 — Service Worker и офлайн-режим

**Что это.** Service Worker — скрипт, который браузер запускает в отдельном фоновом потоке. Он перехватывает сетевые запросы страницы и может отвечать на них из кэша, благодаря чему приложение продолжает работать без интернета.

**Как в проекте.** Регистрация SW выполняется в `client/src/index.js`, сам воркер лежит в `client/public/sw.js`. При установке он кэширует базовые ресурсы приложения, а в обработчике `fetch` использует стратегии кэширования (см. ПР15). Благодаря этому каркас магазина открывается даже при отключённой сети.

---

## ПР14 — Web App Manifest (PWA)

**Что это.** Файл-манифест описывает приложение для браузера: имя, иконки, цвет темы, режим отображения. С ним сайт можно установить на рабочий стол или домашний экран как обычное приложение и запускать в отдельном окне без адресной строки.

**Как в проекте.** Манифест лежит в `client/public/manifest.json` и подключён в `client/public/index.html` через `<link rel="manifest">`. В нём заданы имя «КастрюляМаркет», `display: standalone`, фирменный цвет темы `#6366f1`, тёмный background и набор иконок от 16×16 до 512×512 (включая maskable для Android). В HTML также добавлены мета-теги для корректной интеграции с iOS и Android.

---

## ПР15 — HTTPS и App Shell

**Что это.** Современные браузеры требуют HTTPS для работы Service Worker, Push API и установки PWA. App Shell — архитектурный паттерн: статический каркас приложения (шапка, меню, стили) кэшируется агрессивно (Cache First), а данные всегда берутся из сети (Network First).

**Как в проекте.**

- **HTTPS.** Скрипт `server/setup-https.js` (`npm run cert`) генерирует корневой CA и сертификат для localhost с помощью `node-forge`, затем через `certutil` добавляет CA в доверенные хранилища Windows. При запуске `server/app.js` проверяет наличие `localhost.pem` и поднимает HTTPS-сервер на порту 3000. Клиент работает по HTTPS благодаря `HTTPS=true` и путям к сертификатам в `client/.env`.
- **App Shell.** В `sw.js` статические файлы (`/`, `index.html`, `manifest.json`, иконки) загружаются по стратегии **Cache First** — сразу из кэша. Навигационные запросы идут по **Network First** с фолбэком на `index.html`. Запросы к `/api/*` намеренно пропускаются, чтобы данные товаров и пользователей всегда были свежими.

## ПР16 — WebSocket и Push-уведомления

**Что это.** WebSocket даёт двустороннюю связь в реальном времени между клиентом и сервером — страница получает события без перезагрузки. Push-уведомления работают даже когда вкладка закрыта: сервер через VAPID-ключи отправляет сообщение push-сервису браузера, а тот будит Service Worker и показывает уведомление.

**Как в проекте.**

- **WebSocket (socket.io).** Сервер поднимает `Socket.IO` поверх того же HTTPS-сервера (`server/app.js`, блок после `https.createServer`). Клиент подключается в `client/src/api/socket.js`. Когда админ добавляет новый товар, сервер рассылает событие `productAdded` всем подключённым клиентам — список товаров обновляется мгновенно у всех открытых вкладок.
- **Push.** VAPID-ключи генерируются автоматически при первом запуске сервера и сохраняются в `vapid-keys.json`. Логика подписки — в `client/src/api/push.js` (`subscribeToPush` / `unsubscribeFromPush`), кнопка включения находится в `client/src/components/PushToggle.jsx`. В `sw.js` обработчик `push` показывает уведомление с иконкой магазина. При добавлении нового товара сервер рассылает push всем подписчикам.

## ПР17 — Напоминания с отложенной push-рассылкой

**Что это.** Практическое применение связки WebSocket + Push: пользователь создаёт напоминание с указанным временем, сервер держит его в памяти и ровно в нужный момент отправляет push-уведомление. В уведомлении есть кнопка «Отложить», которая переносит его на 5 минут.

**Как в проекте.** Серверная часть в `server/app.js`: событие `newReminder` через socket.io добавляет запись в `Map`-хранилище `reminders` и ставит `setTimeout` на нужную задержку. Когда таймер срабатывает, `web-push` рассылает уведомление всем подписчикам и эмитится событие `reminderFired` для всех открытых клиентов. Эндпоинт `POST /api/snooze?reminderId=...` сбрасывает текущий таймер и ставит новый на 5 минут. В `sw.js` обработчик `notificationclick` ловит нажатие на кнопку `snooze` и вызывает этот эндпоинт прямо из Service Worker — даже если вкладка магазина закрыта.

## ПР18 — Офлайн-очередь для администратора

**Что это.** Расширение офлайн-режима: при отключённой сети над каталогом товаров отображается предупреждающий баннер. Администратор при этом сохраняет возможность добавлять новые карточки — они сохраняются в `localStorage` и автоматически синхронизируются с сервером при восстановлении подключения.

**Как в проекте.**

- **Обнаружение офлайна.** Хук `useOffline()` в `ProductsPage.jsx` каждые 2 секунды делает HEAD-запрос к `/manifest.json?_=timestamp` с `cache: 'no-store'`, обходя кэш Service Worker. При недоступности сети сразу показывается жёлтый баннер «Проверьте подключение к интернету».
- **Кэш API.** Service Worker теперь кэширует GET-ответы `/api/products` в отдельный кэш `api-cache-v1`. При офлайне возвращает закэшированные данные, чтобы карточки оставались видны.
- **Офлайн-очередь (admin).** Если пользователь — `admin` и сеть отсутствует, новый товар сохраняется в `localStorage` под ключом `pendingProducts`. Карточка отображается в каталоге с меткой «⏳ Офлайн».
- **Автосинхронизация.** Когда сеть восстанавливается, хук сравнивает предыдущее и текущее состояние офлайна. При переходе «офлайн → онлайн» вызывается `syncPendingProducts()`, которая отправляет каждый pending-товар через `api.createProduct()`, очищает очередь и обновляет список.

## Запуск

```bash
# Сервер (первый раз — PowerShell от администратора)
cd server
npm install
npm run cert     # только при первом запуске
npm start

# Клиент
cd client
npm install
npm start
```

Открыть: `https://localhost:3001`
